## Step 1: Specify in-scope stakeholders, tasks, input and output documents

### Inputs:

Requirements for the conceptual model from homework 1, logical schema from homework 2

### **Outputs:**

In scope stakeholders, scoping diagram, list of in scope tasks, input documents, output documents

**In-Scope Stakeholders:**

1. Client
    
    purchases licenses, uses their features, activates them on devices
    
2. License Manager
    
    creates licenses, assigns them to clients, and manages license details
    
3. Support Manager
    
    handles license activation on devices and monitors activation limits
    

**Out of scope:**

1. Software developers
2. Payment Service (some external system)
3. Marketing and sales teams

Scoping

The database system is focused on software license management. The in-scope part includes license creation, license activation, license status checking,  feature assignment
Payment processing is out of scope, but payment confirmation is used as an input for license creation.
Software development, marketing, and sales (teams) are also out of scope because these activities are not automated by the database system

### **Input Documents**

The system will use these input documents : License Request, Payment Confirmation, Device Information, Feature List

And detailed description of each:

**License Request**

contains a request from a client to create a software license

structure : client_id, request_id, request_date, licence_type, duration, activation_limit

**Payment Confirmation**

confirms successful payment

structure : transaction_id, payment_date, payment_status, payment_method, amount, currency, license_id

**Device Information**

contains information about a device where the license is activated

structure : device_id, hardware_id, operating_system, activation_date, license_id

**Feature List**

contains available features that can be assigned to a license

structure : feature_id, feature_name, description

### **Output Documents**

The system generates the following output documents: License Record, Activation Record, License Status Report, Updated License Record

And detailed description of each:

**License Record**

provides information about a created license

structure : license_id, license_key, issue_date, expiration_date , status, client_id, license_type

**Activation Record**

contains information about license activation on a device

structure : activation_id, device_id, license_id, activation_date, activation_status

**License Status Report**

shows the current status of a license

structure : license_id, status, expiration_date, activation_count, activation_limit

## Step 4: Refine database schema

### Inputs:

Refined conceptual model developed in Step 3

**Changes Compared to HW2**

The relational schema was updated based on the refined conceptual model:

- The ternary relation utilizes was removed, therefore the table device_feature is not included in the updated schema.
- A new attribute activation_count was added to the software_license table.
- All other tables remain mostly unchanged.

The relational schema was implemented using SQL DDL queries.

Each table was created using the “CREATE TABLE” statement, including primary keys and foreign keys.

```sql
DROP DATABASE IF EXISTS software_license;
CREATE DATABASE IF NOT EXISTS software_license
    CHARACTER SET 'utf8mb4' COLLATE 'utf8mb4_unicode_ci';
USE software_license;

DROP TABLE IF EXISTS client;
CREATE TABLE client
(
    client_id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT 'Primary key of client',
    contact_name VARCHAR(100) NOT NULL COMMENT 'Client name',
    contact_email VARCHAR(150) NOT NULL COMMENT 'Client contact email',
    street VARCHAR(150),
    city VARCHAR(100),
    country VARCHAR(100),
    postal_code VARCHAR(20),
    PRIMARY KEY (client_id)
) ENGINE = innodb COMMENT = 'Client entity type.';

DESCRIBE client;

CREATE TABLE license_type
(
    type_id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '(PK)' ,
    type_name VARCHAR(50) NOT NULL COMMENT 'Name of the license type',
    description VARCHAR(300) COMMENT 'Description of the license',
    PRIMARY KEY (type_id)
);

DESCRIBE license_type;

CREATE TABLE feature
(
    feature_id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '(PK)',
    feature_name VARCHAR(100) NOT NULL,
    PRIMARY KEY (feature_id)
) ENGINE = innodb COMMENT = 'Feature entity type';

DESCRIBE feature;

CREATE TABLE software_license
(
    license_key VARCHAR(80) NOT NULL COMMENT '(PK)',
    license_id BIGINT UNSIGNED NOT NULL COMMENT 'Internal license identifier',
    issue_date DATE NOT NULL COMMENT 'Time when license was generated',
    status VARCHAR(30) NOT NULL COMMENT 'Current license status',
    duration INT UNSIGNED NOT NULL COMMENT 'Duration of license in days',
    limit_activation INT UNSIGNED NOT NULL COMMENT 'Max allowed activations',
    activation_count INT UNSIGNED NOT NULL DEFAULT 0 COMMENT 'Current number of activations',
    client_id BIGINT UNSIGNED NOT NULL,
    type_id BIGINT UNSIGNED NOT NULL,
    PRIMARY KEY (license_key)
) ENGINE = innodb COMMENT = 'Software license entity type';

CREATE TABLE payment
(
    transaction_id VARCHAR(80) NOT NULL COMMENT 'PK of payment',
    payment_status VARCHAR(30) NOT NULL COMMENT 'Payment status',
    payment_method VARCHAR(30) NOT NULL,
    amount INT UNSIGNED NOT NULL COMMENT 'Payment amount',
    currency VARCHAR(10) NOT NULL COMMENT 'Currency code',
    tax_rate INT UNSIGNED NOT NULL,
    license_key VARCHAR(80) NOT NULL,
    PRIMARY KEY (transaction_id)
) ENGINE = innodb COMMENT = 'Payment entity type';

CREATE TABLE device
(
    license_key VARCHAR(80) NOT NULL COMMENT 'Part of PK and reference to software license',
    hardware_id VARCHAR(100) NOT NULL COMMENT 'Hardware identifier of the device',
    device_status VARCHAR(30) NOT NULL COMMENT 'Current status of the device',
    activation_date DATE NOT NULL COMMENT 'Date when the device was activated',
    PRIMARY KEY (license_key, hardware_id)
) ENGINE = innodb COMMENT = 'Device entity type';

CREATE TABLE software_license_feature
(
    license_key VARCHAR(80) NOT NULL COMMENT 'Part of PK and reference to software license',
    feature_id BIGINT UNSIGNED NOT NULL COMMENT 'Part of PK and reference to feature',
    PRIMARY KEY (license_key, feature_id)
) ENGINE = innodb COMMENT = 'Relation between software license and feature';

ALTER TABLE software_license
```

## Step 5: Normalize your database to 3NF

### Inputs:

Refined relational schema from Step 4

#### **Outputs:**

Database schema normalized to 1NF, 2NF, and 3NF

**Normalization to 1NF**

For normalizing a table to 1NF we check if all the attributes are atomic. Non-atomic attributes are
split into atomic parts.

The tables in the schema satisfy this condition because:

- **client** contains only atomic values such as client name, email, and address fields
- **license_type** contains only atomic values
- **feature** contains only atomic values
- **software_license** contains only atomic values
- **payment** contains only atomic values
- **device** contains only atomic values
- **software_license_feature** contains only atomic key values

Therefore, all relations are in 1NF.

**Normalization to 2NF**

For normalizing a table to 2NF we check if it has partial functional dependencies of non-prime
attributes on prime attributes. If found, partial functional dependencies are removed by table
decomposition.

All tables are in the second normal form.

Tables with a single primary key, such as **client**, **license_type**, **feature**, **software_license**, and **payment**, automatically satisfy 2NF because all attributes depend on the primary key.

For tables with composite keys:

- In **device**, the primary key is (license_key, hardware_id).
In this design, the table represents a device activation record.
The attributes **device_status** and **activation_date** describe the activation and depend on both the license and the device.
- In **software_license_feature**, the primary key is (license_key, feature_id) and there are no additional attributes, so no partial dependency exists.

Therefore, no partial dependencies are present, and the schema is in 2NF.

**Normalization to 3NF**

For normalizing a table to 3NF we check if it has transitive functional dependencies of non-prime
attributes on prime attributes. If found, transitive functional dependencies are removed by table
decomposition.

The schema also satisfies the third normal form.

For example:

- In **software_license**, the attribute **type_id** references the **license_type** table.
The details of the license type (such as name and description) are stored separately, not inside the **software_license** table. This avoids transitive dependencies.
- In **payment**, the attribute **license_key** is a foreign key, but no license-related attributes are duplicated in this table.
- In **client**, all attributes such as **contact_name**, **contact_email**, and address fields depend directly on **client_id** and do not depend on each other.
- In **device**, the non-key attributes **device_status** and **activation_date** describe the device activation record and do not depend on each other.

Therefore, the schema does not contain transitive dependencies and satisfies 3NF.

The relational schema satisfies the requirements of 1NF, 2NF, and 3NF.
No additional decomposition was required.
